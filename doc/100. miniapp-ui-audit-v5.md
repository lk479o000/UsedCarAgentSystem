# 小程序对照 `design-system-v5` 整改总结（阶段一）

更新时间：2026-04-28  
范围：`ucar-miniapp`（小程序）+ 相关后端接口（`ucar-api`）

---

## 背景与目标

你提出的问题集中在两类：
- **合规与可用性**：是否符合微信《小程序内用户帐号登录规范》，以及审核可用性（可取消/拒绝登录、不得强制登录、域名 HTTPS 合法域名）。
- **UI/UX 质量**：当前 UI 未达到 `doc/99. design-system-v5..md` 的玻璃拟态 + 霓虹蓝点缀风格，存在多处功能缺失与交互/视觉不一致。

本阶段策略：
- **先修功能**（避免“看着再美也用不了”）。
- **再做高影响 UI**（底部导航、首页概览、线索/结算列表、线索详情关键弹层、线索编辑表单、个人中心头像比例）。
- 保持上线形态不变：**正式版仍以微信一键登录为主**，只在开发/体验环境提供账号密码登录入口用于联调。

## 已完成（本阶段落地）

### A. 登录合规/体验调整

1) **取消启动即强制跳登录**
- 移除 `ucar-miniapp/app.js` 里启动时全局 `checkLogin()` 强跳登录页行为。
- 现在：tabBar 页面可游客进入；敏感页按需跳转登录（见下）。

2) **游客模式（不调业务接口）**
- `首页/线索/结算/我的` 未登录时展示“游客模式”提示 + “去登录”按钮。
- 未登录时页面不会调用业务接口；仅显示占位/空态。

3) **敏感页按需强制登录**
- 详情/表单类页面改为 `checkLogin({ redirect: true })`：未登录直接跳登录页。
- 涉及页面：`leads/detail`、`leads/form`、`followups/form`、`settlements/detail`、`settlements/form` 等。

4) 登录页增加 **“暂不登录”** 次级入口并优化美观
- 从纯文本改为次级胶囊按钮，并做垂直居中（flex）。
- 入口位置调整为卡片下方，避免与主登录按钮争夺视觉权重。

### B. 仅开发/体验可用：账号密码登录（不影响正式版）

目的：你在域名 HTTPS 未生效、微信一键登录联调受阻时，仍可在开发/体验环境进入系统。

- 仅 `envVersion !== 'release'` 时显示“账号密码登录”入口
- 接口：
  - `GET /api/v1/auth/captcha`
  - `POST /api/v1/auth/login`
- 小程序端：登录页新增折叠面板（账号/密码/验证码），登录成功写入 token/userInfo。

### C. 功能修复（你点名的硬问题）

1) **线索姓名搜索无效 → 已修复**
- 根因：后端 `/api/v1/user/lead` 原先不支持 `customer_name/phone`；小程序传 `keyword` 不生效。
- 处理：
  - 后端 `/api/v1/user/lead` 支持 `customer_name`、`customer_phone`
  - 小程序输入自动映射：纯数字→手机号，否则→姓名

2) **结算无客户筛选 → 已补齐（前后端）**
- 后端：
  - 管理员：`GET /api/v1/settlement` 支持 `customer_name/phone`（通过 include Lead where like）
  - 经纪人：`GET /api/v1/user/settlement` 支持 `customer_name/phone`
- 小程序：结算页新增搜索框并传参

3) **线索列表看不出买/卖 → 已补齐**
- 线索列表增加 `买/卖` 标签。
- 同时补齐 `tag-info` 样式为全局可复用。

### D. UI 改造（对照 `design-system-v5` 的高影响区域）

1) **底部菜单太普通 → 已改为自定义 TabBar（玻璃拟态 + 图标 + 选中态）**
- 原生 `tabBar` 样式可定制很有限，本阶段改用 `custom-tab-bar` 自定义组件。
- 视觉：玻璃卡片、圆角、发光、选中态 dot + 图标高亮。

2) **首页统计占比太大 / 图标仅颜色变化 → 已优化**
- 统计项：
  - 每项使用不同图标（总线索/成交/失败/待结算/已结算）
  - 缩小 padding/字号
  - 最后一项横跨两列，减少“压屏”

3) **线索 tab 文本未居中 → 已修复**
- 筛选 tab 改为 inline-flex 并设置固定高度，保证文字垂直居中。

4) **线索详情：更新状态/成交/失败弹层不美观 → 已改为玻璃拟态 ActionSheet + Dialog**
- ActionSheet、成交/失败输入弹窗统一外观、按钮层级、点击反馈。
- 同时处理自定义 tabBar 下页面底部留白避免遮挡。

5) **编辑线索不美观 → 已统一表单控件**
- `leads/form` 表单：label、必填、input/picker/textarea/按钮全部统一为一套玻璃浅底 + 轻描边风格。
- 同样处理底部留白避免被自定义 tabBar 遮挡。

6) **我的页头像过大 → 已缩小**
- 头像容器从 `w-16 h-16` 调整为 `w-12 h-12`，更贴合微信头像实际观感。

## 未做到（本阶段未覆盖/未完全达到）与原因

### 1) 设计系统“组件化沉淀”未完成（仍有页面各写各的 WXSS）
**现状**：虽然 `app.wxss` 已有 `card-glass/btn/tag` 等基础，但表单控件、弹层、列表 cell 仍分散在各页面 WXSS。  
**影响**：后续改一处风格要改多处，容易不一致。  
**原因**：本阶段优先交付“可用 + 高影响页面变好看”，未集中做基础组件库封装。

### 2) 线索详情“基本信息区”仍偏“表格行”，整体仍可更精致
**现状**：信息展示是多行 key/value 的边框分割，视觉还不够“卡片化/信息层级化”。  
**可改方向**：改成“标题区（姓名+标签+电话）+ 信息 chips（买/卖、状态、车型）+ 备注块 + 时间/归属”。

### 3) 结算详情/结算表单 UI 未统一到新表单体系
**现状**：结算相关页面仍未按新的表单/弹层体系统一。  
**影响**：线索链路与结算链路视觉割裂，整体不“同一套设计系统”。  
**原因**：本阶段优先修复“缺少筛选”与线索链路的高频页面。

### 4) TabBar 图标资源体系未做到“品牌级定制”
**现状**：自定义 TabBar 使用 `i-lucide-*` 图标类（依赖 unocss 图标生成）。  
**风险**：若 unocss 编译链路缺失/不稳定，图标可能不生效；且难以实现品牌定制图标。  
**可行方案**：
- **方案 A（继续用 unocss）**：补齐/固化 unocss 编译流程与 CI 校验，确保图标稳定。
- **方案 B（不依赖 unocss）**：落地一套静态 tabbar icons（selected/unselected 的 PNG/SVG），自定义 TabBar 用 `<image/>` 渲染。

### 5) 性能项（骨架屏/虚拟列表/分包）未做
`design-system-v5` 提到骨架屏、虚拟列表、分包加载，本阶段未实现。  
**原因**：先保证“功能正确 + UI 统一 + 审核风险下降”，再做性能优化更稳。

### 6) 微信手机号能力链路未升级到新口径
当前仍使用 `getPhoneNumber` 的 `encryptedData/iv` + `session_key` 解密方式。  
更推荐链路：`e.detail.code` → 服务端调用 `getuserphonenumber` 换取手机号。  
**原因**：本阶段聚焦登录规范“不得强制登录”与 UI，未做协议级升级。

## 下一阶段（可做到的方案与优先级）

### P0（建议先做）—— 全局组件样式沉淀到 `app.wxss`
目标：减少页面私有 WXSS，所有页面统一满足 `design-system-v5` 风格。

- **表单控件**：`.form-card/.form-item/.form-label/.form-control/.form-textarea/.form-submit`
- **弹层**：`.sheet-*`、`.dialog-*`
- **列表 cell**：`.list-card/.list-title/.list-subtitle/.list-meta`

落地方式：
- 先把现在线索表单与详情弹层（已做的一版 page 样式）上提到全局
- 再迁移结算表单/详情、绑定手机、登录账号面板

### P0 —— 线索详情“基本信息区”重排版
方案：做“头部信息卡”增强层级：
- 姓名 + 买/卖标签 + 状态标签
- 电话/车型/时间 chips
- 备注使用浅背景块，归属经纪人（管理员）更突出

涉及：`pages/leads/detail.wxml/.wxss`

### P1 —— 结算详情/结算表单 UI 统一
把结算表单改为 `.form-*` 体系；结算详情做“金额卡片 + 状态/时间 chips”。

涉及：`pages/settlements/detail.*`、`pages/settlements/form.*`

### P1 —— 线索列表 UI 细化
本阶段已做：tab 垂直居中、买/卖标签、搜索生效。  
下一步建议：
- 搜索栏增加 clear 按钮、输入防抖、空态文案优化
- 卡片信息层级：姓名/标签一行；车型/时间一行；管理员视角经纪人信息更突出

涉及：`pages/leads/leads.wxml/.wxss/.js`

### P2 —— unocss 编译链路完善（若继续使用 `i-lucide-*`）
`design-system-v5` 提到 `unocss-preset-weapp`，需要确认项目具备并可重复执行：
- `ocss.config.js`、`npm run unocss`
- 产物 `unocss.wxss` 生成并被 `app.wxss` 引入（当前已 `@import './unocss.wxss'`，但需固化生成流程）

若不想依赖编译：切到静态图标资源渲染（见“未做到 4)”的方案 B）。

### P2 —— 性能优化
- 首页/列表骨架屏
- 列表虚拟化（数据量大时）
- 分包加载（按 home/leads/settlements/profile 拆）
