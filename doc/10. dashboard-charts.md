## 二、服务器资源约束与决策

### 2.1 当前部署资源分配（2C2G阿里云）

| 服务 | 内存限制 | 实际占用 | CPU限制 |
|------|---------|---------|---------|
| MySQL 8.4 | 900m | ~700-800m (buffer_pool 256M) | 1.0核 |
| Node API | 350m | ~200-256m (V8堆限制256M) | 0.8核 |
| Nginx | 120m | ~30-50m | 0.3核 |
| 系统+Docker | ~400m | OS内核+Docker守护进程 | - |
| **合计** | **~1770m** | **~1400-1500m** | **2.1核** |

剩余可用内存：**约500-600m**

### 2.2 Redis决策：不上Redis

**结论：2C2G服务器不建议部署Redis。**

原因：
- 剩余500-600m内存，Redis最少需200-300m才稳定，几乎没有余量
- MySQL复杂查询或连接数飙升时，极易OOM
- 统计数据变化频率低（分钟级），不需要Redis的毫秒级响应

**替代方案：Node.js内存缓存**

使用 `node-cache` 或简单的 `Map` + TTL，5分钟过期，效果等同Redis，零额外内存开销：

```javascript
const NodeCache = require('node-cache');
const statsCache = new NodeCache({ stdTTL: 300, checkperiod: 60 });

async function getDashboardData(key, queryFn) {
  const cached = statsCache.get(key);
  if (cached) return cached;
  const data = await queryFn();
  statsCache.set(key, data);
  return data;
}
```

## 四、页面架构：首页 + 大屏分离

### 4.1 为什么要分离

| 对比 | 首页 | 大屏 |
|------|------|------|
| 定位 | 快速概览 + 操作入口 | 深度数据分析 |
| 图表数 | 2-4个 | 6-10个 |
| 接口数 | 1个（聚合） | 2-3个（聚合） |
| 加载速度 | 秒开 | 可接受2-3秒 |
| 交互 | 轻量，点击跳转 | 丰富，筛选下钻 |
| 使用频率 | 每次登录 | 需要分析时 |

### 4.2 页面结构

```
首页（Dashboard）
├── 指标卡片行（5个KPI）
├── 2个核心图表（趋势 + 分布）
└── [进入大屏] 按钮 → 大屏页面（Analytics）

大屏页面（Analytics）
├── 时间筛选器（7天/30天/90天）
├── 第一排：线索趋势 + 状态分布
├── 第二排：转化漏斗 + 成交额趋势
├── 第三排：经纪人排行 + 失败原因
└── 第四排（管理员专属）：地区分布 + 结算概览
```

## 五、接口设计：聚合方案

### 5.1 核心原则：少接口，后端聚合

**问题**：如果每个图表一个接口，首页+大屏需要10+个接口，2C2G服务器扛不住。

**方案**：合并为2-3个聚合接口，后端一次查询返回所有数据。

### 5.2 接口清单

| 接口 | 页面 | 调用时机 | 返回数据 |
|------|------|---------|---------|
| `GET /dashboard/overview` | 首页 | 页面加载 | 指标卡片 + 趋势图 + 状态分布（1次请求搞定首页） |
| `GET /dashboard/analytics?range=30` | 大屏 | 进入大屏 | 全部图表数据（趋势、分布、漏斗、排行、原因、地区、结算） |
| `GET /dashboard/agent-stats?range=30` | 大屏(经纪人) | 进入大屏 | 经纪人个人维度全部图表数据 |

### 5.4 后端缓存策略

```
请求 → 检查内存缓存 → 命中 → 返回（<1ms）
                    → 未命中 → 查询MySQL → 写入缓存(5分钟TTL) → 返回
```

| 缓存Key | TTL | 说明 |
|---------|-----|------|
| `dashboard:overview:{userId}` | 5分钟 | 首页数据，按用户隔离（经纪人看个人数据） |
| `dashboard:analytics:admin:{range}` | 5分钟 | 管理员大屏数据，按时间范围隔离 |
| `dashboard:analytics:agent:{userId}:{range}` | 5分钟 | 经纪人大屏数据 |

## 六、首页图表配置策略

### 6.1 决策：简单化（前端配置数组），不做可视化配置后台

**选择简单化的理由：**

| 维度 | 分析 |
|------|------|
| 用户规模 | 管理员1-2人 + 经纪人若干，总共不到10人 |
| 角色数 | 仅2个（管理员/经纪人） |
| 图表总数 | 管理员6-8个，经纪人4个，排列组合有限 |
| 服务器 | 2C2G资源紧张，配置系统额外开销不划算 |
| 运维 | 无专职运维，改一行配置重新部署比维护配置系统更简单 |

**配置化的成本远大于收益：**
- 需新建配置表、配置管理页面、图表缩略图预览、动态渲染引擎、配置读取API
- 这套工作量可能比做图表本身还大
- 管理员想换图表，改一行代码重新部署，5分钟搞定

### 6.2 实现方式：前端配置数组

```javascript
const HOME_CHARTS = [
  { key: 'leadTrend', title: '线索趋势', visible: true, order: 1 },
  { key: 'statusDistribution', title: '线索状态分布', visible: true, order: 2 },
  // { key: 'dealTrend', title: '成交额趋势', visible: false },
  // { key: 'conversionFunnel', title: '转化漏斗', visible: false },
]
```

调整首页图表：改 `visible` 和 `order`，重新构建部署即可。

### 6.3 未来配置化路线（如需要）

等业务跑起来、用户反馈多了，可升级为轻量配置方案：
- 新建 `c_dashboard_config` 表，存储用户角色的首页图表配置
- 管理后台增加"首页配置"页面，用图表名称+小图标勾选（不做可视化预览，太重）
- 经纪人不可配置，只有管理员能配
- 当前阶段不实施，预留接口即可

## 八、技术实现要点

### 8.1 图表库选型

推荐 **ECharts**，理由：
- 功能全面，支持所有图表类型（含漏斗图）
- 中文文档完善，社区活跃
- 按需引入，体积可控

### 8.2 前端路由

```
/dashboard      → 首页（轻量概览）
/analytics      → 大屏（深度分析）
```

### 8.3 性能保障

| 措施 | 说明 |
|------|------|
| 接口聚合 | 首页1个接口，大屏1个接口，避免10+并发请求 |
| 后端内存缓存 | node-cache，5分钟TTL，零额外内存开销 |
| 后端聚合计算 | SQL层面GROUP BY聚合，不传原始数据给前端 |
| 懒加载 | 大屏页面按需加载ECharts（动态import） |
| 时间范围限制 | 默认30天，最长90天，避免大范围查询 |

### 8.4 MySQL查询优化

当前MySQL配置（2G生存版）：
- `max_connections = 60`
- `innodb_buffer_pool_size = 256M`
- `tmp_table_size = 32M`

统计查询优化要点：
- 所有统计字段均有索引（`status`, `user_id`, `created_at`, `city_id`）
- GROUP BY 查询走索引，不会产生全表扫描
- 限制时间范围（最长90天），避免大范围聚合
- 聚合接口加缓存，5分钟内相同查询直接返回

## 九、可行性评估

### 9.1 全部图表均可基于现有数据实现

| 评估项 | 结论 |
|--------|------|
| 线索趋势 | ✅ `created_at` 有索引，GROUP BY DATE查询高效 |
| 状态分布 | ✅ `status` 有索引，直接COUNT |
| 成交额趋势 | ✅ `car_actual_price` 字段 + `updated_at` 可用 |
| 转化漏斗 | ✅ 6个状态值(0-5)直接统计各阶段数量 |
| 经纪人排行 | ✅ `user_id` 有索引，关联 `c_user` 取姓名 |
| 失败原因 | ✅ `fail_reason` 在status=5时必填 |
| 地区分布 | ✅ `city_id` 有索引，`c_region` 表可关联 |
| 客户类型 | ✅ `customer_type` 两个值直接COUNT |
| 热门车型 | ⚠️ 可行但 `car_brand` 为自由文本，需数据清洗/归并 |
| 结算概览 | ✅ `c_settlement` 表字段完整 |
| 跟进活跃度 | ✅ `c_lead_followup` 表有 `created_at` 索引 |
| 成交周期 | ✅ `created_at` → `updated_at`(status=4) 计算差值 |
| 响应时效 | ✅ 首条跟进记录的 `followup_time` - `c_lead.created_at` |

### 9.2 需注意的数据质量问题

- **car_brand / car_model**：自由文本输入，可能存在"奔驰"和"梅赛德斯-奔驰"等不一致，建议后续增加品牌字典表或模糊匹配
- **car_price / car_actual_price**：允许 NULL 和默认值0，统计时需过滤无效数据
- **city_id**：允许 NULL，统计地区分布时需排除无地区数据的记录
