# 开发规则与约定

## 技术栈
- 后端：Node.js + Express（统一服务，供管理后台和小程序使用）
- PC端前端（管理后台）：Vue3
- 小程序：微信小程序原生框架
- 数据库：MySQL 8+

## 命名规范
- 文件夹：小写-中划线（如 ucar-api、ucar-admin-web）
- 变量：camelCase（如 customerName）
- 数据表：snake_case（如 leads、agents）
- 接口路径：统一前缀 `/api/v1/...`

## 提交规范
- Git commit message 格式：`feat: xxx` / `fix: xxx` / `doc: xxx`
- 分支策略：`main`（稳定版）、`dev`（开发版）、`feature-xxx`（功能分支）

## 代码风格
- 使用 ESLint + Prettier 保持统一风格
- 缩进统一为 2 空格
- 使用单引号 `'`，避免双引号
- 所有函数必须有注释，说明输入输出
- 禁止使用 `console.log`，统一使用日志库

## 错误处理
- 所有接口返回统一 JSON 格式：
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```
- 错误统一返回：
```json
{
  "code": 1,
  "message": "error detail"
}
```
- 日志记录错误堆栈，按日期切割存储

## 安全规则
- 所有接口需鉴权（JWT）
- 管理员接口与经纪人接口分离，避免越权
- 登录接口需验证码或密码校验，防止暴力破解
- 数据库连接信息存放在 .env 文件，不可硬编码

## 性能约定
- 接口响应时间 < 200ms（常规查询）
- 数据库表需建立必要索引（如 phone、status 字段）
- 避免 N+1 查询，使用 JOIN 或批量查询

## 部署规则
- 后端必须支持 Docker 部署
- 日志统一写入文件并按日期切割
- 使用环境变量区分开发/生产环境
- 云服务器选择轻量化配置，减少成本

## API 版本管理
- 所有接口统一前缀 `/api/v1/`
- 后续迭代时可升级为 `/api/v2/...`

## 测试与质量保证
- 单元测试覆盖率 ≥ 70%
- 提交前必须通过 ESLint 检查
- 所有接口需有至少一个测试用例

## 文档维护规则
- 当数据表结构发生变化时，必须更新 `doc/3. database-schema.md` 文件
- 当码表（c_dict）数据发生变化时，必须更新 `sql/初始化码表.sql` 文件
- 当数据表结构或业务逻辑发生变化时，必须更新 `sql/常用sql.sql` 文件
- 所有文档修改都应该保持一致性，确保不同文档之间的信息不冲突
- 文档更新后，相关的SQL文件也应该同步更新
- 新增功能或修改功能时，应同时更新相关的文档
