# 开发规则与约定

## 技术栈
- **后端**：Node.js + Express（统一服务，供管理后台和小程序使用）
- **PC端前端（管理后台）**：Vue3
- **小程序**：微信小程序原生框架
- **数据库**：MySQL 8+
- **样式**：
  - PC端：UnoCSS
  - 小程序：unocss-preset-weapp（原子化样式）
- **图标**：Lucide 图标集

## 命名规范
- **文件夹**：小写-中划线（如 ucar-api、ucar-admin-web）
- **变量**：camelCase（如 customerName）
- **常量**：UPPER_SNAKE_CASE（如 STATUS_MAP）
- **数据表**：snake_case（如 c_user、c_lead）
- **接口路径**：统一前缀 `/api/v1/...`
- **小程序页面**：kebab-case（如 leads-detail、settlements-form）
- **小程序组件**：kebab-case（如 lead-card、status-tag）

## 提交规范
- **Git commit message 格式**：`type(scope): description`
  - `type`：feat（新功能）、fix（修复）、docs（文档）、style（样式）、refactor（重构）、test（测试）、chore（构建）
  - `scope`：模块名称（如 lead、settlement、auth）
  - `description`：简洁明了的描述
- **分支策略**：
  - `main`：稳定版
  - `dev`：开发版
  - `feature-xxx`：功能分支
  - `fix-xxx`：修复分支

## 代码风格
- **通用规则**：
  - 使用 ESLint + Prettier 保持统一风格
  - 缩进统一为 2 空格
  - 使用单引号 `'`，避免双引号
  - 所有函数必须有注释，说明输入输出
  - 禁止使用 `console.log`，统一使用日志库
- **小程序特有**：
  - 使用原子化样式，减少自定义 CSS
  - 页面逻辑清晰，避免过度嵌套
  - 组件化开发，复用公共逻辑
  - 合理使用 `setData`，避免频繁更新

## 错误处理
- **接口返回**：
  - 成功：
  ```json
  {
    "code": 0,
    "message": "success",
    "data": {}
  }
  ```
  - 失败：
  ```json
  {
    "code": 1,
    "message": "error detail"
  }
  ```
- **日志记录**：
  - 错误堆栈按日期切割存储
  - 关键操作记录详细日志
- **小程序错误处理**：
  - 网络请求统一错误处理
  - 页面加载错误捕获
  - 用户操作错误提示

## 安全规则
- **通用安全**：
  - 所有接口需鉴权（JWT）
  - 管理员接口与经纪人接口分离，避免越权
  - 登录接口需验证码或密码校验，防止暴力破解
  - 数据库连接信息存放在 .env 文件，不可硬编码
- **小程序安全**：
  - 敏感信息加密存储
  - 接口调用频率限制
  - 手机号脱敏处理
  - 密码强度检测

## 性能约定
- **后端**：
  - 接口响应时间 < 200ms（常规查询）
  - 数据库表需建立必要索引（如 phone、status 字段）
  - 避免 N+1 查询，使用 JOIN 或批量查询
- **小程序**：
  - 下拉刷新 + 上拉加载优化
  - 图片资源懒加载
  - 页面渲染性能优化
  - 网络请求合并与缓存

## 部署规则
- **后端**：
  - 支持 Docker 部署
  - 日志统一写入文件并按日期切割
  - 使用环境变量区分开发/生产环境
  - 云服务器选择轻量化配置，减少成本
- **小程序**：
  - 代码包大小控制在 2MB 以内
  - 发布前进行安全扫描
  - 配置合法的域名和隐私政策

## API 版本管理
- 所有接口统一前缀 `/api/v1/`
- 后续迭代时可升级为 `/api/v2/...`
- 版本兼容性保证

## 测试与质量保证
- **后端**：
  - 单元测试覆盖率 ≥ 70%
  - 提交前必须通过 ESLint 检查
  - 所有接口需有至少一个测试用例
- **小程序**：
  - 功能测试覆盖所有页面
  - 兼容性测试（不同微信版本）
  - 性能测试（启动时间、页面切换）

## 文档维护规则
- **数据库文档**：
  - 当数据表结构发生变化时，必须更新 `doc/3. database-schema.md` 文件
  - 当码表（c_dict）数据发生变化时，必须更新 `sql/初始化码表.sql` 文件
- **SQL文件**：
  - 当数据表结构或业务逻辑发生变化时，必须更新 `sql/常用sql.sql` 文件
- **API文档**：
  - 新增或修改接口时，必须更新 `doc/4. api-spec.md` 文件
- **小程序文档**：
  - 新增或修改功能时，必须更新 `doc/9. 小程序功能规划方案.md` 文件
  - 完成开发后，必须更新 `doc/9. 小程序功能开发进度.md` 文件
- **一致性要求**：
  - 所有文档修改都应该保持一致性，确保不同文档之间的信息不冲突
  - 文档更新后，相关的SQL文件也应该同步更新
  - 新增功能或修改功能时，应同时更新相关的文档

## 小程序开发规范
- **目录结构**：
  ```
  ucar-miniapp/
  ├── pages/              # 页面
  │   ├── leads/          # 线索管理
  │   │   ├── leads       # 线索列表
  │   │   ├── detail      # 线索详情
  │   │   └── form        # 线索表单
  │   ├── settlements/    # 结算管理
  │   └── profile/        # 个人中心
  ├── utils/              # 工具函数
  ├── images/             # 图片资源
  ├── static/             # 静态资源
  ├── app.js              # 应用入口
  ├── app.json            # 全局配置
  ├── app.wxss            # 全局样式
  ├── uno.config.js       # UnoCSS 配置
  └── unocss.wxss         # 生成的原子化样式
  ```
- **样式规范**：
  - 使用 `unocss-preset-weapp` 原子化样式
  - 语义化命名（如 `card-glass`、`btn-primary`）
  - 统一颜色变量
  - 响应式设计（使用 rpx 单位）
- **图标规范**：
  - 使用 `@unocss/preset-icons` + Lucide 图标集
  - 图标命名统一（如 `i-lucide-user`、`i-lucide-car`）
- **权限控制**：
  - 根据角色动态显示/隐藏操作按钮
  - 后端接口数据权限验证
  - 前端UI权限控制
