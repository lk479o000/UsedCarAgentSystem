## 一、现状分析

### 1.1 现有页面

| 页面 | 路径 | 功能 |
|------|------|------|
| 线索列表 | `pages/leads/leads` | 状态折叠筛选 + 搜索 + 卡片列表 + FAB新增 |
| 线索详情 | `pages/leads/detail` | 基本信息卡片 + 跟进记录列表 + 底部操作栏 |
| 线索表单 | `pages/leads/form` | 新增/编辑线索，表单字段平铺 |
| 跟进表单 | `pages/followups/form` | 新增/编辑跟进记录 |

### 1.2 核心痛点

1. **跟进记录缺乏时间轴感**：当前为简单卡片堆叠（`bg-gray-50 rounded-xl p-4 mb-3`），无法直观感受跟进节奏和时间跨度
2. **详情页信息层级扁平**：状态、价格等关键信息不够突出，所有信息平铺在一个卡片内
3. **经纪人无法看到跟进记录**：跟进记录区域加了 `wx:if="{{isAdmin}}"`，经纪人视角下完全不可见
4. **状态流转缺乏可视化**：6个状态仅用一个标签展示，无法直观感知当前进度和后续路径
5. **列表页缺少跟进摘要**：需进入详情才能看到跟进动态，无法在列表快速判断优先级
6. **操作栏占用空间大**：底部固定操作栏遮挡内容，且操作项过多（编辑/更新状态/结算/删除）
7. **列表页缺少区域筛选**：已有区域闭包表（`c_region` + `c_region_cl`）和线索区域字段（`provinceId`/`cityId`/`districtId`），但列表页无区域筛选入口，无法按省/市/区快速定位线索

### 1.3 数据模型回顾

**线索状态流转**：
```
待跟进(0) → 跟进中(1) → 已看车(2) → 已报价(3) → 已成交(4)
                                           ↘ 已失败(5)
```

**跟进记录字段**：
- `followup_content` — 跟进内容（必填）
- `followup_result` — 跟进结果（可选）
- `followup_time` — 这次跟进时间（必填）
- `next_followup_time` — 下次跟进时间（可选）
- `operator_user_id` — 操作人（关联 User）

**线索区域字段**：
- `province_id` — 所属省份ID（关联 `c_region`）
- `city_id` — 所属城市ID（关联 `c_region`）
- `district_id` — 所属区县ID（关联 `c_region`）

**区域闭包表**（`c_region_cl`）：
- `region_id` — 当前节点ID
- `ancestor_id` — 祖先节点ID
- `path_length` — 路径长度（0=自身，1=父节点...）
- `full_path` — 完整路径（如"广东省 > 深圳市 > 南山区"）
- `is_direct` — 是否直接父子关系
- `is_root_node` — 祖先是否为根节点

**区域API**（已实现）：
- `GET /api/v1/region/provinces` — 查询所有省份
- `GET /api/v1/region/cities/:provinceId` — 根据省份ID查城市
- `GET /api/v1/region/districts/:cityId` — 根据城市ID查区县
- `GET /api/v1/region/search` — 根据名称模糊搜索

## 二、移动端空间约束分析

> 本章对每个设计决策进行空间约束审查，确保方案在真实手机屏幕上可行。

### 2.1 屏幕尺寸基准

| 设备 | 逻辑宽度 | 占比参考 |
|------|---------|---------|
| iPhone SE | 375px | 小屏基准（最窄） |
| iPhone 15 | 393px | 主流基准 |
| iPhone 15 Pro Max | 430px | 大屏基准 |
| 安卓主流 | 360-412px | 需兼容 |

**关键约束**：
- 最小可用宽度：360px（安卓低端机）
- 触摸热区最小：44px × 44px（Apple HIG）/ 48dp × 48dp（Material）
- 小程序导航栏高度：44px（自定义）/ 48px（原生）
- 底部安全区：34px（iPhone X 及以上）

### 2.2 逐项空间审查

#### 问题1：7个状态Tab横向排布

**空间计算**：
- 7个Tab文字：`全部(42)` `待跟进(5)` `跟进中(12)` `已看车(8)` `已报价(6)` `已成交(9)` `已失败(2)`
- 每个Tab最小宽度 = 文字宽度 + 两侧间距 ≈ 60-80px
- 7个Tab总宽度 ≈ 490-560px
- 屏幕可用宽度 ≈ 360-430px
- **结论：必然超出屏幕，需要横向滚动**

**风险**：
- 用户可能不知道右侧还有更多Tab，漏看「已成交」「已失败」
- 横向滚动在列表页顶部，与纵向滚动列表存在手势冲突
- 频繁切换Tab时需要来回滚动，操作成本高

**解决方案对比**：

| 方案 | 空间占用 | 操作步骤 | 风险 |
|------|---------|---------|------|
| A. 横向滚动Tab | 1行，需滚动 | 1步点击 | 漏看右侧Tab |
| B. 双行Tab | 2行固定，无滚动 | 1步点击 | 占用纵向空间约80px |
| C. 主次分组Tab | 1行核心+折叠更多 | 1-2步 | 需展开次级 |
| D. 下拉筛选（保留现有） | 1行折叠 | 2步（展开+选择） | 步骤多但空间省 |

**⭐ 推荐方案C：主次分组Tab**（详见2.3节）

---

#### 问题2：状态进度条5节点+标签

**空间计算**：
- 5个节点：待跟进 → 跟进中 → 已看车 → 已报价 → 已成交
- 每个节点：圆点(8px) + 标签(3字×14px≈42px) + 间距 ≈ 70px
- 5节点总宽度 ≈ 350px + 4条连线 ≈ 380px
- **结论：375px屏幕上刚好挤下，但标签文字过小、节点过密**

**风险**：
- 「待跟进」「已成交」等3字标签在70px空间内字号需≤12px，可读性差
- 节点间距过小，手指难以精确点击
- 已失败状态分支无法在水平进度条中表达

**解决方案对比**：

| 方案 | 空间占用 | 信息量 | 风险 |
|------|---------|-------|------|
| A. 水平5节点+缩写 | 1行≈360px | 完整但拥挤 | 标签过小 |
| B. 水平5节点+无标签 | 1行≈200px | 需用户记忆节点含义 | 不直观 |
| C. 竖向步骤条 | 2-3行高度 | 完整清晰 | 占纵向空间 |
| D. 当前状态突出+可流转态 | 1行≈200px | 精简聚焦 | 丢失全局进度 |

**⭐ 推荐方案D：当前状态突出显示**（详见3.2节）

---

#### 问题3：快捷操作按钮5个横排

**空间计算**：
- 5个按钮：拨打、跟进、状态、编辑、更多
- 每个按钮最小触摸区：60px（图标+文字+间距）
- 5按钮总宽度 ≈ 300px + 4×间距 ≈ 360px
- **结论：375px勉强放下，但按钮过小、间距过紧**

**风险**：
- 按钮间距过小，容易误触
- 视觉拥挤，缺乏呼吸感
- 不同状态显示不同按钮组合，布局不稳定

**解决方案对比**：

| 方案 | 空间占用 | 操作步骤 | 风险 |
|------|---------|---------|------|
| A. 5按钮横排 | 1行≈360px | 1步 | 拥挤误触 |
| B. 3主+更多菜单 | 1行≈240px | 1-2步 | 平衡 |
| C. 底部固定2主+更多 | 固定底部 | 1-2步 | 遮挡内容 |
| D. 纵向操作列表 | 多行 | 1步 | 占纵向空间 |

**⭐ 推荐方案B：3主按钮+更多菜单**（详见3.2节）

---

#### 问题4：列表卡片增加跟进摘要

**空间计算**：
- 现有卡片高度 ≈ 120-140px
- 增加跟进摘要：分割线(1px) + 摘要行(20px) + 下次跟进(20px) + 内边距 ≈ 50px
- 新卡片高度 ≈ 170-190px
- **结论：卡片增高约35%，首屏可见卡片数从4-5个减少到3-4个**

**风险**：
- 首屏信息密度降低，浏览效率下降
- 滚动距离增加
- 并非所有线索都有跟进摘要，有/无摘要的卡片高度不一致，视觉跳动

**解决方案对比**：

| 方案 | 卡片高度 | 首屏卡片数 | 信息量 |
|------|---------|-----------|-------|
| A. 完整摘要区 | +50px | 3-4个 | 最多 |
| B. 仅1行摘要 | +25px | 3-4个 | 适中 |
| C. 无摘要，保持现状 | 不变 | 4-5个 | 最少 |
| D. 紧凑摘要（嵌入meta区） | +0px | 4-5个 | 少但无额外空间 |

**⭐ 推荐方案B：仅1行摘要**（详见2.3节）

---

#### 问题5：详情页总长度

**空间计算**（375px宽屏，667px高 iPhone SE）：
- 导航栏：44px
- 状态头部卡片：200px（含进度条+基本信息）
- 快捷操作区：50px
- 跟进时间轴（3条记录）：350px
- 更多信息区：120px
- 底部安全区：34px
- **总高度 ≈ 798px，超过1屏，需滚动**

**风险**：
- 关键操作（跟进、状态更新）可能不在首屏
- 滚动过长导致用户疲劳
- 经纪人视角内容少，但管理员视角内容很多

**解决方案**：
- 管理员视角：首屏展示状态+操作+最新1条跟进，其余跟进和时间轴在下方
- 经纪人视角：首屏即可看完所有内容
- 使用锚点/Tab切换，而非无限纵向滚动

### 2.3 约束审查结论

| 设计项 | 原方案 | 约束问题 | 调整方案 |
|--------|--------|---------|---------|
| 状态筛选 | 7个横向滚动Tab | 必然溢出，漏看 | 主次分组Tab |
| 状态进度条 | 水平5节点+标签 | 拥挤不清晰 | 当前状态突出+可流转态 |
| 快捷操作 | 5按钮横排 | 误触风险 | 3主按钮+更多菜单 |
| 跟进摘要 | 完整摘要区 | 卡片过高 | 仅1行摘要 |
| 详情页长度 | 全部纵向排列 | 超过1屏 | 分区+首屏聚焦 |
| 区域筛选 | 无 | 无法按区域定位线索 | 搜索行内嵌区域按钮+半屏Picker |

## 三、线索列表页改造方案

### 3.1 布局结构

```
┌─────────────────────────────────┐
│  [跟进中(12)] [已看车(8)] [更多▼]│  ← 主次分组Tab
│  全部:42  待跟进:5  已报价:6     │  ← 更多下拉
│  已成交:9  已失败:2              │
├─────────────────────────────────┤
│  📍 区域 ▼  🔍 搜索客户姓名/手机号│  ← 区域筛选+搜索
├─────────────────────────────────┤
│  ┌───────────────────────────┐  │
│  │ 张三  [买家] [跟进中]      │  │
│  │ 宝马 3系  📍朝阳  🕐2h前   │  │
│  │ 📋 客户表示下周看车        │  │  ← 1行跟进摘要
│  └───────────────────────────┘  │
│                                 │
│  ┌───────────────────────────┐  │
│  │ 李四  [卖家] [已报价]      │  │
│  │ 奥迪 A6  📍海淀  🕐昨天    │  │
│  │ 📋 已发送报价单            │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

### 3.2 改造要点

#### 3.2.1 ⭐ 主次分组Tab（替代7个横向滚动Tab）

**设计思路**：将7个状态分为「高频核心」和「低频更多」两组。

**核心Tab（始终可见，1行3-4个）**：
```
[跟进中(12)]  [已看车(8)]  [更多▼]
```

- 核心Tab选择依据：取当前线索数量最多的2-3个状态动态显示
- 「更多」按钮点击后弹出下拉面板，显示全部状态：

```
┌─────────────────────────────────┐
│  [跟进中(12)]  [已看车(8)]  [更多▼]│
│  ┌───────────────────────────┐  │
│  │ 全部(42)                   │  │  ← 下拉面板
│  │ 待跟进(5)                  │  │
│  │ 已报价(6)                  │  │
│  │ 已成交(9)                  │  │
│  │ 已失败(2)                  │  │
│  └───────────────────────────┘  │
└─────────────────────────────────┘
```

**规则**：
- 核心Tab动态计算：取线索数量最多的2个非终态状态 + 「更多」
- 如果用户从「更多」中选择了某个状态，该状态临时替换核心Tab中数量最少的那个
- 「全部」始终在「更多」面板的第一项
- 选中态：主色背景 + 白色文字；未选中：浅灰背景 + 灰色文字

**空间计算**：
- 3个核心Tab：每个约100-120px，总宽300-360px ✅ 适配所有屏幕
- 下拉面板：覆盖在列表上方，不占额外空间 ✅

**对比原方案（7个横向滚动Tab）**：

| 维度 | 7个横向滚动 | 主次分组Tab |
|------|-----------|-----------|
| 首屏可见状态数 | 3-4个（需滚动看其余） | 2个核心+5个下拉 |
| 操作步骤 | 1步（但可能需先滚动） | 核心1步，次级2步 |
| 漏看风险 | 高（右侧Tab不可见） | 低（下拉面板一次全展示） |
| 空间占用 | 1行固定 | 1行固定+临时下拉 |

#### 3.2.2 卡片增加1行跟进摘要

**现状**：卡片仅显示客户名、车辆、区域、时间。

**改为**：在现有信息下方增加1行跟进摘要（不增加分割线，不增加下次跟进时间行，保持紧凑）。

```
┌─────────────────────────────────┐
│ 张三  [买家] [跟进中]  138****8  │  ← 第1行：姓名+标签+电话
│ 宝马 3系  📍朝阳  🕐2小时前     │  ← 第2行：车辆+区域+时间
│ 📋 客户表示下周看车             │  ← 第3行：跟进摘要（新增）
└─────────────────────────────────┘
```

**空间计算**：
- 新增1行 ≈ 20px + 8px上下间距 = 28px
- 原卡片高度约120px → 新高度约148px
- 首屏卡片数：4-5个 → 4个 ✅ 可接受

**规则**：
- 跟进摘要最多显示1行，超出截断（`text-overflow: ellipsis`）
- 无跟进记录时不显示该行，卡片保持原高度
- 超期跟进（下次跟进时间已过）：摘要行文字变红色
- 管理员卡片额外显示经纪人信息（保持现有逻辑）

#### 3.2.3 超期跟进提醒

下次跟进时间已过但状态仍为「待跟进/跟进中」的线索：

- 卡片左侧加4px红色竖条
- 跟进摘要行文字变为红色
- 不额外增加空间

```
▌ 张三  [买家] [跟进中]  138****8
▌ 宝马 3系  📍朝阳  🕐2小时前
▌ ⚠️ 下次跟进已超期：昨天 14:00     ← 红色文字
```

#### 3.2.4 搜索框简化

**现状**：经纪人和客户两个独立搜索框，占2行约100px。

**改为**：单个搜索框，输入内容同时匹配客户姓名和手机号。

```
┌─────────────────────────────────┐
│ 🔍 搜索客户姓名/手机号          │  ← 1行搜索框
└─────────────────────────────────┘
```

- 管理员模式下，搜索框右侧增加「经纪人」筛选按钮，点击弹出经纪人选择面板
- 搜索框高度约44px，节省约56px纵向空间

#### 3.2.5 ⭐ 区域筛选（利用区域闭包表）

**现状**：线索已有 `provinceId`/`cityId`/`districtId` 字段，列表卡片也展示了 `regionText`，但无区域筛选入口。

**改为**：搜索框左侧增加区域筛选按钮，点击弹出省市区三级联动选择器。

**筛选入口**（搜索行内嵌）：
```
┌─────────────────────────────────────────┐
│  📍 区域 ▼  │  🔍 搜索客户姓名/手机号   │
└─────────────────────────────────────────┘
```

- 区域按钮宽度约80px，搜索框占剩余空间
- 未选择区域时显示「区域 ▼」，选择后显示区域名称如「深圳 ▼」
- 区域名称过长时截断，如「深圳市 > 南... ▼」

**区域选择器**（点击后弹出半屏Picker）：
```
┌─────────────────────────────────┐
│  选择区域              [确定]   │
├─────────────────────────────────┤
│  ┌──────┬──────┬──────┐        │
│  │ 广东省│ 深圳市│ 南山区│        │  ← 三列滚动选择
│  │ 北京市│      │      │        │
│  │ 上海市│      │      │        │
│  └──────┴──────┴──────┘        │
│                                 │
│  ☐ 包含下级区域                 │  ← 利用闭包表查询所有后代
│                                 │
│  [重置]            [确定]       │
└─────────────────────────────────┘
```

**「包含下级区域」功能**：
- 利用闭包表的 `getRegionAllDescendants(ancestorId)` 方法，查询选中区域的所有后代节点ID
- 例如选择「深圳市」并勾选「包含下级」，则筛选 `city_id IN (深圳市及所有区县的ID集合)`
- 不勾选时，仅精确匹配选中的层级（选区县匹配 `district_id`，选城市匹配 `city_id`，选省份匹配 `province_id`）

**空间计算**：
- 筛选入口：与搜索框同行，不额外占用纵向空间 ✅
- 选择器：半屏弹出（约350px高），不占页面空间 ✅
- 三列滚动选择器宽度：375px / 3 ≈ 125px/列，每列可显示5-6个选项 ✅

**筛选逻辑**：
- 仅选省份 → 按 `province_id` 筛选
- 选省份+城市 → 按 `city_id` 筛选
- 选省份+城市+区县 → 按 `district_id` 筛选
- 勾选「包含下级」→ 利用闭包表后代查询，按 `province_id`/`city_id`/`district_id` IN 筛选
- 管理员和经纪人均可使用区域筛选

| 适用场景 | 线索少 | 线索多、跨区域经营 |

| 交互 | 说明 |
|------|------|
| 下拉刷新 | 原生下拉刷新 + 骨架屏占位 |

### 3.3 API适配

列表接口需返回跟进摘要字段（需后端配合）：

```
GET /api/v1/lead 或 /api/v1/user/lead

响应新增字段：
- lastFollowupContent: string   — 最近一次跟进内容（截断至30字）
- nextFollowupTime: string      — 最近一次跟进记录的下次跟进时间
- statusCounts: object          — 各状态数量统计（可选，用于Tab数量显示）
```

后端实现方式：在 `getLeadList` 查询时，关联查询最新一条跟进记录，提取 `followup_content` 和 `next_followup_time`。`statusCounts` 可单独接口或在首次请求时附带返回。

**区域筛选参数**（需后端新增）：

```
GET /api/v1/lead 或 /api/v1/user/lead

请求新增参数：
- province_id: number     — 按省份筛选
- city_id: number         — 按城市筛选
- district_id: number     — 按区县筛选
- region_ids: string      — 包含下级时，逗号分隔的区域ID列表（利用闭包表后代查询结果）
```

后端实现方式：
1. 在 `routes/lead.js` 的 filters 中新增 `provinceId`/`cityId`/`districtId`/`regionIds` 参数
2. 在 `leadService.getLeadList` 的 where 条件中增加区域筛选：
   - 精确匹配：`where.provinceId = filters.provinceId`
   - 包含下级：`where[Op.or] = [{ provinceId: { [Op.in]: regionIds } }, { cityId: { [Op.in]: regionIds } }, { districtId: { [Op.in]: regionIds } }]`
3. 闭包表后代查询已有现成方法：`regionService.getRegionAllDescendants(ancestorId)`

1. 5节点+3字标签在375px宽度内拥挤不堪
2. 已失败是分支状态，水平进度条无法自然表达
3. 用户最关心的是「现在在哪个阶段」和「接下来能去哪」，而非完整流程全景
4. 完整流程全景在列表页的Tab筛选中已经体现

**空间计算**（375px宽屏）：

- 间距：16rpx = 8px
- 内容卡片可用宽度：375 - 24(卡片内边距) - 12(节点) - 8(间距) - 32(卡片内边距) = 299px ✅
- 每条记录高度：约80-120px（视内容长度）
- 3条记录 + 标题 + 添加按钮 ≈ 400px ✅ 可在1屏内展示

#### 5.1.2 智能折叠规则

|-----------|---------|---------|

- 3条记录约300px，加上头部卡片(200px)和操作区(50px)，首屏总高约550px
- 在iPhone SE(667px)上，首屏还能看到「更多信息」的标题 ✅

```css
  --text-primary: #0F172A;
  --text-secondary: #475569;
  --text-tertiary: #94A3B8;
  --text-placeholder: #CBD5E1;
}
```

**使用方式**：`var(--primary, #0EA5E9)` — 变量优先，硬编码作为fallback。

### 11.2 圆角规范

| 组件类型 | 圆角值 | 类名约定 |
|---------|--------|---------|
| 页面级卡片（card-glass） | 16rpx | `.card-glass` |
| 表单卡片（form-card） | 28rpx | `.form-card` |
| 列表卡片（list-card） | 28rpx | `.list-card` |
| 按钮-主要 | 28rpx | `.btn-primary` |
| 按钮-幽灵 | 28rpx | `.btn-ghost` |
| 按钮-胶囊 | 48rpx | `.btn-pill` |
| 标签/Chip | 12rpx | `.tag-*` / `.chip` |
| 时间轴内容卡片 | 12rpx | `.timeline-card` |
| 输入框 | 12rpx | `.form-control` |
| 对话框 | 28rpx | `.dialog` |

**禁止**在页面级WXSS中随意覆盖圆角值。如需特殊圆角，应定义新的语义化类名。

### 11.4 图标规范

| 级别 | 尺寸 | 使用场景 |
|------|------|---------|
| icon-sm | 24rpx | 标签内、chip内、时间轴编辑图标 |
| icon-md | 32rpx | 列表meta区、搜索图标、筛选图标 |
| icon-lg | 40rpx | 快捷操作区、FAB按钮 |
| icon-xl | 48rpx | 空状态图标、统计图标 |

**图标颜色**：统一通过 `color: currentColor` 继承父元素颜色，禁止在图标元素上直接设置 `color`。

**图标与文字间距**：统一使用 `gap: 8rpx`（chip内）或 `gap: 12rpx`（按钮内），禁止使用 `margin` 控制间距。

### 11.5 间距规范

| 变量 | 值 | 使用场景 |
|------|-----|---------|
| `--space-1` | 8px (16rpx) | 图标与文字间距、chip内间距 |
| `--space-2` | 16px (32rpx) | 卡片内边距、表单项间距 |
| `--space-3` | 24px (48rpx) | 区域间距、分组间距 |
| `--space-4` | 32px (64rpx) | 页面内边距 |
| `--space-5` | 48px (96rpx) | 大区域间距 |

**原则**：间距使用 `--space-*` 变量或 `rpx` 值，禁止使用 `px` 值（除特殊场景如1px边框）。
